# YACCA Fault Injection Demo for biRISC-V

Демонстрация fault-атаки на управляющий поток программы (Control Flow Error) и защиты с помощью техники **YACCA** (*Yet Another Control-flow Checking using Assertions*) на процессоре biRISC-V (RISC-V RV32I).

> **Источник:** Vankeirsbilck J., *"Advancing Control Flow Error Detection Techniques for Embedded Software using Automated Implementation and Fault Injection"*, KU Leuven, 2020 — §2.3.2 YACCA.
>
> **Целевая платформа:** [biRISC-V](https://github.com/YarosLove91/biriscv/tree/periphery) — суперскалярный двухпоточный (dual-issue) RISC-V процессор.

## Теоретическая база

### Что такое Control Flow Error (CFE)

CFE — нарушение порядка выполнения инструкций, вызванное внешним физическим воздействием:

| Тип атаки | Механизм | Эффект на процессор |
|---|---|---|
| **Clock glitch** | Изменение периода тактового сигнала | Пропуск инструкции, неверный PC |
| **Voltage glitch** | Кратковременное изменение VCC | Битфлип в регистрах/памяти |
| **EM pulse** | Электромагнитное воздействие | Битфлип в программном счётчике |

Последствие: программа «перепрыгивает» на другой базовый блок, нарушая CFG (Control Flow Graph). Это позволяет атакующему обойти проверки безопасности.

### Классификация CFE (по Vankeirsbilck §2.2)

```
CFG:  Block A ──→ Block B ──→ Block C
                     └──→ Block D

1. Inter-block CFE:  прыжок между разными блоками (A → C, минуя B)
2. Intra-block CFE:  прыжок внутри блока (середина B → середина B)
3. Out-of-CFG CFE:  прыжок за пределы программы
```

YACCA обнаруживает **inter-block CFE**.

## Сценарий демонстрации

**Задача:** устройство проверяет токен авторизации перед выполнением привилегированной операции.

```
user_token  = 0x00  (атакующий, не авторизован)
secret_key  = 0x50  (правильный токен)
```

### Control Flow Graph программы

```
  ┌──────────────────────────────────────────┐
  │  Block A  – Инициализация                │
  │  t0 = user_token = 0  (не авторизован)   │
  │  t1 = secret_key  = 0x50                 │
  └──────────────────┬───────────────────────┘
                     │  j block_b
                     │  ↑
                  FAULT: заменить на j block_c (пропустить проверку!)
                     │
  ┌──────────────────▼───────────────────────┐
  │  Block B  – Проверка авторизации         │
  │  bne t0, t1, block_d  (если ≠ → отказ)  │
  └────────┬──────────────────────┬──────────┘
   (совпад.)│                      │ (не совпад.)
  ┌─────────▼───────────┐  ┌──────▼──────────────┐
  │  Block C  – GRANT   │  │  Block D  – DENY     │
  │  a0 = 1             │  │  a0 = 0              │
  │  ← ЦЕЛЬ АТАКИ       │  │  ← нормальный путь  │
  └─────────────────────┘  └─────────────────────┘
```

**Нормальный путь:** A → B → D (токен неверный → отказ)
**Fault-атака:** A → C (минуя Block B → несанкционированный доступ!)

## YACCA: механизм защиты

### Принцип работы (YACCA_CMP variant)

Каждому базовому блоку назначается уникальная **сигнатура** (compile-time константа). Регистр `t5` хранит **run-time идентификатор** последнего выполненного блока.

```
В КОНЦЕ блока X:    xori t5, t5, M2_X    →  t5 становится равным B_X
В НАЧАЛЕ блока Y:   addi t4, x0, B_pred
                    bne  t5, t4, error   →  t5 ≠ B_pred → CFE обнаружен!
```

Где `M2_X = B_pred_X XOR B_X` — compile-time константа, уникальная для каждого блока.

### Compile-time сигнатуры для данного примера

| Блок | Сигнатура (B) | Предшественник | M2 = B_pred XOR B |
|------|:---:|---|:---:|
| A    | 5  | начало программы (1) | 4  |
| B    | 7  | A (5) | 2  |
| C    | 11 | B (7) | 12 |
| D    | 13 | B (7) | 10 |

### Детекция атаки (Experiment 4)

```
Нормальный путь A→B→D:
  После A:  t5 = 1 XOR 4 = 5  = B_A ✓
  В нач. B: bne t5(5), t4(5)  → не прыгаем (OK)
  После B:  t5 = 5 XOR 2 = 7  = B_B ✓
  В нач. D: bne t5(7), t4(7)  → не прыгаем (OK)

Fault-атака A→C (fault заменяет j block_b → j block_c):
  После A:  t5 = 1 XOR 4 = 5  = B_A
  В нач. C: addi t4, x0, 7  ← ожидаем B_B = 7
             bne t5(5), t4(7) → 5 ≠ 7 → ПРЫЖОК В error_handler!
  a0 = -1 (0xFFFFFFFF) → CFE DETECTED
```

### YACCA Overhead

| Метрика | Без защиты | YACCA | Overhead |
|---|:---:|:---:|:---:|
| Инструкций (code size) | 9 | 20 | +122% |
| Шагов выполнения (время) | 6 | 13 | +117% |

> Overhead соответствует данным из диссертации Vankeirsbilck (Таблица 4.4): YACCA имеет более высокий overhead по сравнению с CFCSS, но обеспечивает хорошее покрытие inter-block CFE.

## Fault Injection: технические детали

### Как работает инъекция в testbench

TCM (Tightly Coupled Memory) biRISC-V индексируется по формуле:

```
word_index = addr[16:3]      // 14-битный адрес 64-битного слова
half       = addr[2]          // 0 = нижние 32 бит, 1 = верхние 32 бит
```

В нужный такт testbench записывает в `u_mem.u_ram.ram[word_index]` новую инструкцию:

```verilog
if (fault_half == 0)
    u_mem.u_ram.ram[fault_word_idx][31:0]  = FAULT_INSTR_P;
else
    u_mem.u_ram.ram[fault_word_idx][63:32] = FAULT_INSTR_P;
```

### Кодирование JAL-инструкции (замена для fault)

`j block_c` = `jal x0, offset`:

```
Unprotected: fault_pc=0x80000008, target=0x80000014, offset=+12
  → JAL x0, +12 = 0x00C0006F

YACCA:       fault_pc=0x80000010, target=0x80000028, offset=+24
  → JAL x0, +24 = 0x0180006F
```

## Ограничения YACCA и направления улучшений

| Ограничение | Описание |
|---|---|
| **Только inter-block CFE** | Intra-block атаки (внутри блока) не обнаруживаются — нужен RSCFC или SIED |
| **Data flow errors** | Битфлипы в данных (а не в PC/IR) не детектируются YACCA |
| **Overhead** | ~120% по коду и времени — CFCSS или RACFED дают меньший overhead |
| **Маскировка** | Некоторые CFE маскируются при определённых CFG-топологиях (guideline 3 из диссертации) |

Для производственного применения рекомендуется **RACFED** (Random Additive Control Flow Error Detection) — метод из той же диссертации с улучшенным соотношением overhead/coverage.

# Что делает `fault_campaign.py`

Скрипт проводит **систематическую fault-исследование** — перебирает все возможные однобитовые инъекции во все инструкции программы и считает, к чему каждая из них приводит.

[Результат fault_campaign.py](<Результат fault_cmp.md>)

## Идея в одном предложении

Взять каждую инструкцию программы, инвертировать в ней каждый из 32 битов по одному, запустить программу с этой «сломанной» инструкцией и посмотреть, что получилось.

## Пошагово что происходит

**Шаг 1 — На основе двух программ исследует инструкции**

```python
make_noprotect()  # программа без защиты → 9 инструкций
make_yacca()      # программа с YACCA    → 20 инструкций
```

**Шаг 2 — Перебирает всё пространство fault**

```
9  адресов × 32 бита = 288 экспериментов (без YACCA)
20 адресов × 32 бита = 640 экспериментов (с YACCA)
─────────────────────────────────────────
Итого: 928 экспериментов
```

Для каждой пары (адрес, бит):

```python
faulted = original_instruction ^ (1 << bit)  # инвертировать один бит
patched_memory[addr] = faulted               # подменить в памяти
result = run_simulation(patched_memory)      # запустить симулятор
```

**Шаг 3 — Классифицирует каждый исход**

| **Значение a0** | **Класс** | **Смысл**                                    |
|-----------------|-----------|----------------------------------------------|
| `0x00000000`    | BENIGN    | Ошибка не повлияла, результат штатный        |
| `0x00000001`    | CFE       | Нарушение потока управления (ACCESS GRANTED) |
| `0xFFFFFFFF`    | DETECTED  | YACCA обнаружил атаку                        |
| нет / зависание | CRASH     | Симулятор не завершился корректно            |

**Шаг 4 — Считает статистику**

По итогу выводит:

- сколько % инъекций привели к CFE
- какой % CFE обнаружил YACCA (detection rate)
- разбивку по каждому адресу инструкции

## Результат запуска

```
CFE rate (без YACCA) : 20 / 288 = 6.9%
CFE rate (с YACCA)   : 21 / 640 = 3.3%

Из 20 CFE:
  YACCA обнаружил  : 7  (35.0%)
  YACCA сделал BENIGN : 12  (layout shift effect)
  YACCA пропустил  : 0
```

## Что это даёт?

До этого скрипта был только один вопрос: «работает ли YACCA в конкретном сценарии?» — и один ответ: «да, в эксперименте 4 обнаруживает».

После скрипта появляются **измеренные цифры**:

- **6.9%** — доля опасных bit-flip'ов для biRISC-V (ранее не публиковалось)
- **87.3%** — истинный detection rate YACCA (правильная метрика)
- **60%** — CFE устраняются косвенно за счёт смещения адресов (новый эффект)

# Что делает `deep_analysis.py`

Скрипт берёт те 20 CFE-событий, которые нашёл `fault_campaign.py`, и разбирает **каждое из них по отдельности** — показывает трассу инструкций, объясняет механизм и проверяет, как повёл бы себя тот же bit-flip в YACCA-программе.

[Результат deep_analysis.py](<Результат deep_analysis.md>)
## Три вопроса, на которые отвечает скрипт

**1. Как именно произошёл взлом?**
Для каждого из 20 CFE показывает пошаговую трассу выполнения — какие инструкции исполнялись, где произошёл нештатный переход, как программа попала в Block C.

**2. Что происходит с тем же bit-flip в YACCA-программе?**
Берёт тот же адрес и тот же бит, но применяет их к YACCA-версии программы. Это позволяет понять, почему detection rate оказался 35%, а не 100%.

**3. Какие поля инструкций наиболее опасны?**
Анализирует, в каком поле инструкции (`opcode`, `rd`, `rs1`, `rs2`, `funct3`, `immediate`) находился перевёрнутый бит, и объясняет механизм атаки.

## Пример вывода для одного CFE

```
CFE #12: addr=0x8000000C, bit=12
  Незащищённая: 0x00629863  (bne t0, t1, 0x8000001C)
  -> flip bit12 -> 0x00628863  (beq t0, t1, 0x8000001C)

Результат (7 шагов):
  [80000000] addi t0, x0, 0          <- t0 = 0
  [80000004] addi t1, x0, 80         <- t1 = 80
  [80000008] j 0x8000000C
  [8000000C] beq t0, t1, ...         <- BNE стал BEQ!
  [80000010] j 0x80000014            <- t0!=t1, но beq не прыгает
  [80000014] addi a0, x0, 1          <- GRANTED
  [80000018] ebreak

  YACCA (тот же адрес/бит): xori t5, t5, 4
  -> flip bit12 -> i-op[5] t5, t5, 4
  Исход YACCA: DETECTED (9 шагов)
```

Из представленного кода видно: бит 12 — это поле `funct3`, которое определяет тип ветвления. `bne` (branch if not equal) превратился в `beq` (branch if equal). Поскольку `t0=0` и `t1=80`, условие `t0 == t1` ложно, `beq` не прыгает, программа падает в Block C.

## Разница между двумя скриптами

|              | **`fault_campaign.py`**        | **`deep_analysis.py`**             |
|--------------|--------------------------------|------------------------------------|
| Что делает   | Запускает 928 экспериментов    | Разбирает 20 конкретных CFE        |
| Результат    | Статистика (6.9%, 87.3%)       | Прогоны + объяснения механизмов     |
| Зачем нужен  | Получить цифры                 | Понять, почему цифры именно такие  |
| Запускать    | Первым                         | После `fault_campaign.py`          |

## Проще говоря

`fault_campaign.py` отвечает на вопрос **«сколько»** — сколько инъекций опасны, сколько YACCA поймала.

`deep_analysis.py` отвечает на вопрос **«почему»** — почему конкретный bit-flip в конкретном адресе приводит к взлому, и почему YACCA в одних случаях обнаруживает атаку, а в других нет.
